# REST Controller Code Review Checklist

## Категория 1: API Design (Дизайн API)

### 1. ***Неправильные HTTP методы (CRITICAL)***

*   **Проблема:** Метод `getInvitees()` использует `@PostMapping("/getInvitees")`.
*   **Почему это плохо:** POST предназначен для создания ресурсов или отправки данных, изменяющих состояние сервера. Получение данных — **идемпотентная** и **безопасная** операция, для которой предназначен GET. Использование POST нарушает семантику HTTP, сбивает с толку клиентов API и делает невозможным кэширование ответов.
*   **Рекомендация:** Использовать `@GetMapping("/invitees")`.

### 2. ***Неправильные статус коды (CRITICAL)***

*   **Проблема:** Во всех методах возвращается **200 OK**, даже если операция завершилась иначе (создание, удаление, ошибка "не найдено").
*   **Почему это плохо:** REST использует HTTP статус-коды для информирования клиента о результате запроса. Возврат 200 для всех случаев лишает клиента этой важной информации.
   *   **create():** При успешном создании должен возвращаться код **201 Created** с заголовком `Location`, указывающим на новый ресурс.
   *   **delete():** При успешном удалении должен возвращаться код **204 No Content** (без тела ответа).
   *   **getById():** Если объект не найден, должен возвращаться код **404 Not Found**, а не `null`.
*   **Рекомендация:** Использовать `ResponseEntity` для точного контроля над HTTP ответом.

### 3. ***Плохой naming: глаголы в URL (MAJOR)***

*   **Проблема:** URL `@PostMapping("/getInvitees")` содержит глагол "get".
*   **Почему это плохо:** В RESTful API URL должны описывать **ресурсы (существительные)**, а действия над ними определяются HTTP методами.
*   **Рекомендация:** Использовать `/invitees` для всех операций, связанных с коллекцией приглашенных.

### 4. ***Entity вместо DTO в response (CRITICAL)***

*   **Проблема:** Все методы напрямую возвращают JPA сущность `Invitee`.
*   **Почему это плохо:**
   *   **Связность (Coupling):** Внутренняя структура БД становится частью публичного API. Любое изменение в сущности сломает API для клиентов.
   *   **Безопасность:** Можно случайно **экспонировать чувствительные данные** (например, хэш пароля, если он появится).
   *   **Избыточность:** API возвращает поля, не нужные клиенту, или lazy-загрузка может вызвать исключения при сериализации.
*   **Рекомендация:** Создать отдельные **DTO (Data Transfer Objects)** для ответов API и маппить на них сущности.

### 5. ***Нет пагинации для списков (MAJOR)***

*   **Проблема:** Метод `getInvitees()` возвращает `List<Invitee>`.
*   **Почему это плохо:** При миллионе записей попытка загрузить их все в память приведет к **OutOfMemoryError** на сервере, огромной нагрузке на сеть и плохому UX.
*   **Рекомендация:** Добавить параметры для пагинации (`page`, `size`) и возвращать страницу с данными, используя `Pageable` и `Page<InviteeDTO>`.

## Категория 2: Security (Безопасность)

### 1. ***SQL injection через конкатенацию (CRITICAL)***

*   **Проблема:** В методе `create()` формируется SQL запрос конкатенацией: `"SELECT ... WHERE email = '" + email + "'"`.
*   **Почему это плохо:** Злоумышленник может передать в поле `email` значение `' OR '1'='1`, изменив логику запроса, что приведет к утечке, модификации или удалению данных. Это одна из самых опасных уязвимостей.
*   **Рекомендация:** Никогда не конкатенировать строки для SQL. Использовать **параметризованные запросы** (Spring Data JPA, `@Query` с параметрами) или `JdbcTemplate` с плейсхолдерами.

### 2. ***Нет валидации входных данных (CRITICAL)***

*   **Проблема:** Методы принимают `Map<String, Object>` и не проводят валидацию.
*   **Почему это плохо:**
   *   **Целостность данных:** Можно передать `null` и сохранить некорректные данные в БД.
   *   **Неявный контракт:** Использование `Map` скрывает ожидаемую структуру запроса от клиента.
*   **Рекомендация:** Создать DTO-классы (например, `InviteeCreateRequest`) с аннотациями Bean Validation (`@NotBlank`, `@Email`) и использовать `@Valid` в контроллере.

### 3. ***Missing authorization checks (CRITICAL)***

*   **Проблема:** В контроллере нет никаких проверок прав доступа.
*   **Почему это плохо:** Любой, у кого есть доступ к API, может создавать, удалять и изменять приглашенных.
*   **Рекомендация:** Использовать Spring Security и аннотации `@PreAuthorize` для ограничения доступа к методам по ролям.

## Категория 3: Error Handling (Обработка ошибок)

### 1. ***Пустые catch блоки (MAJOR)***

*   **Проблема:** В методе `updateStatus()` блок `catch (Exception e) { return null; }` полностью "съедает" исключение.
*   **Почему это плохо:**
   *   **Потеря информации:** Невозможно узнать, что пошло не так.
   *   **Сложность отладки:** Клиент получает `null` с кодом 200, что крайне запутывает.
*   **Рекомендация:** Никогда не оставлять `catch` блок пустым. Минимум — **логировать ошибку**. Лучше — пробрасывать исключение в `@ControllerAdvice`.

### 2. ***500 на бизнес-ошибки (MAJOR)***

*   **Проблема:** При невалидном статусе выбрасывается `new RuntimeException("Invalid status")`.
*   **Почему это плохо:** `RuntimeException` возвращает статус **500 Internal Server Error** (проблема сервера), хотя проблема на стороне клиента (некорректные данные).
*   **Рекомендация:** Создать свои исключения (например, `InvalidStatusException`) и обрабатывать их в `@ControllerAdvice`, возвращая клиенту статус **400 Bad Request**.

### 3. ***Нет логирования ошибок (MAJOR)***

*   **Проблема:** Ни одно исключение не логируется.
*   **Почему это плохо:** Невозможно мониторить работоспособность приложения и расследовать инциденты.
*   **Рекомендация:** Использовать **SLF4j** для логирования ошибок на уровне сервисов и в `@ControllerAdvice`.

## Категория 4: Code Quality (Качество кода)

### 1. ***Бизнес-логика в контроллере (MAJOR)***

*   **Проблема:** Контроллер содержит логику:
   *   Ручное создание и заполнение объекта `Invitee`.
   *   Валидация статуса.
   *   Формирование SQL запроса.
*   **Почему это плохо:** Нарушает **принцип единственной ответственности (SRP)**. Контроллер должен только обрабатывать HTTP и делегировать задачи сервисному слою (`@Service`). Это усложняет тестирование и поддержку.
*   **Рекомендация:** Вынести всю бизнес-логику в отдельные сервисы.

### 2. ***Дублирование кода (MAJOR)***

*   **Проблема:** Логика поиска `Invitee` по ID дублируется в нескольких методах.
*   **Почему это плохо:** Усложняет поддержку. Изменения придется вносить в нескольких местах.
*   **Рекомендация:** Вынести поиск в **отдельный метод сервиса**.

### 3. ***Hardcoded values (MINOR)***

*   **Проблема:** Строки статусов `"ACTIVE"` и `"INACTIVE"` "захардкожены" в коде.
*   **Почему это плохо:** При изменении списка статусов придется искать и менять строки по всему проекту.
*   **Рекомендация:** Вынести значения в `enum` (например, `InviteeStatus`) или в конфигурационный файл.

### 4. ***Другие проблемы***

*   **`@Autowired` на поле:** Внедрение через конструктор предпочтительнее — оно делает зависимости явными и облегчает тестирование.
*   **Возврат `null`:** Возврат `null` из методов контроллера — плохая практика. Это может привести к `NullPointerException` при сериализации. Лучше возвращать `ResponseEntity` с соответствующим статусом (например, **404 Not Found**).
